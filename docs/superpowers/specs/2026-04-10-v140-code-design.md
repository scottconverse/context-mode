# context-mode v1.4.0 Code Changes — Design Spec

**Date:** 2026-04-10
**Project:** context-mode v1.4.0 (Spec B — code changes)
**Status:** Approved

---

## Overview

Two features plus version bump. No breaking changes.

1. **Learner miss-detection** — Detect when Claude searches for content that was compressed away and finds nothing. Use miss signals to increase retention weights (less aggressive compression for that tool pattern).
2. **ctx_execute rate limiting** — Sliding window throttle to prevent tight execution loops, plus duplicate command detection.

---

## Feature 1: Learner Miss-Detection

### Problem

The learner only tracks retrieval hits (ctx_search finds compressed content). It cannot detect misses (ctx_search returns nothing for content that was compressed away). Without miss detection, the feedback loop is one-sided — it can detect over-compression via hits but cannot detect it via failed searches.

### Design

**PostToolUse miss signal:**

When PostToolUse detects a `ctx_search` call that returns 0 results or all results score below a relevance threshold, write a miss signal file:

```
{signalDir}/miss-{timestamp}.json
```

Format:
```json
{
  "type": "miss",
  "queries": ["the search terms"],
  "timestamp": 1712793600000,
  "resultCount": 0
}
```

**processSignals() extension:**

Extend the existing signal processing to handle miss signals:

1. Read `miss-*.json` files alongside `retrieval-*.json`
2. For each miss signal, compare query words against `content_preview` in recent `compression_log` entries (same 2-word overlap heuristic as hits)
3. If a match is found, mark the decision as `was_missed = 1`
4. Delete the signal file after processing

**Schema migration (v2):**

Add column to `compression_log`:
```sql
ALTER TABLE compression_log ADD COLUMN was_missed INTEGER DEFAULT 0;
```

This is migration v2. The migration runner backs up the DB before applying.

**Weight calculation:**

Current:
```javascript
const retrievalRate = row.retrieved / row.total;
const retentionScore = Math.min(retrievalRate * RETENTION_MULTIPLIER, 1.0);
```

New:
```javascript
const retrievalRate = row.retrieved / row.total;
const missRate = row.missed / row.total;
const signalRate = retrievalRate + missRate;
const retentionScore = Math.min(signalRate * RETENTION_MULTIPLIER, 1.0);
```

Both hits and misses increase retention — both indicate the compressed content was needed.

**getWeights() query update:**

```sql
SELECT
  COUNT(*) as total,
  SUM(CASE WHEN was_retrieved = 1 THEN 1 ELSE 0 END) as retrieved,
  SUM(CASE WHEN was_missed = 1 THEN 1 ELSE 0 END) as missed
FROM compression_log
WHERE tool_pattern = ? AND timestamp > ?
```

**ctx_stats update:**

Add miss rate to the Learner section:
```
## Learner
  Patterns tracked:  47 decisions
  Retrieval rate:    8/47 (17.0%)
  Miss rate:         3/47 (6.4%)
  Confidence:        High
```

**getLifetimeStats() update:**

Return `totalMisses` alongside `totalRetrievals`.

## Feature 2: ctx_execute Rate Limiting

### Problem

No guard against Claude calling `ctx_execute` in a tight loop (e.g., retrying a failing script). Wastes tokens and compute.

### Design

**Sliding window throttle:**

Mirror the existing `ctx_search` throttle pattern in `server/index.js`:

| Calls in 60s window | Behavior |
|---------------------|----------|
| 1–5 | Normal execution |
| 6–10 | Warning appended to output |
| 11+ | Blocked with error message |

Constants:
```javascript
const EXECUTE_WINDOW_MS = 60000;
const EXECUTE_WARN_AFTER = 5;
const EXECUTE_BLOCK_AFTER = 10;
```

Warning message (appended to normal output):
```
[context-mode] High execution rate (N calls in 60s). Consider batching with ctx_batch_execute.
```

Block message (returned as error):
```
BLOCKED: {N} ctx_execute calls in {X}s. Use ctx_batch_execute to combine operations, or wait for the throttle window to reset.
```

**Duplicate command detection:**

Track a rolling buffer of recent command hashes (SHA-256 of `language + code`). If the exact same code is submitted within 60s, return immediately:

```
Duplicate execution detected — identical code was just run {N}s ago. Use ctx_search to find the previous result, or modify the code before re-running.
```

Buffer size: last 20 commands. Entries expire after 60s.

**Scope:** Applies to `ctx_execute` only. `ctx_batch_execute` and `ctx_execute_file` are excluded — batch operations are already the recommended alternative, and file processing is naturally varied.

## Testing

### Learner miss-detection tests (in test/learner.test.js)
- Miss signal file is written by PostToolUse when ctx_search returns 0 results
- processSignals() reads and processes miss signal files
- Miss signal matches against compression_log content_preview
- was_missed column populated correctly
- Weight calculation incorporates miss rate
- getLifetimeStats() returns totalMisses
- Schema migration v2 adds was_missed column
- Existing data preserved across migration

### Rate limiting tests (in test-e2e.js or new test file)
- Normal execution within threshold (1–5 calls)
- Warning appended after EXECUTE_WARN_AFTER
- Blocked after EXECUTE_BLOCK_AFTER
- Duplicate command detection returns immediately
- Window resets after 60s
- ctx_batch_execute and ctx_execute_file are not throttled

### ctx_stats tests
- Miss rate displayed in Learner section
- Stats accurate with both hits and misses
